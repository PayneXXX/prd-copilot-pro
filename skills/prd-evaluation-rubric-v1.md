# AI PRD 评估 Rubric · v1

## 📖 这份文档是什么

这份 Rubric 是为身处 AI 浪潮中的产品经理（尤其是 Agent 开发者与算法产品经理）准备的一把“度量衡”。

传统的 PRD 习惯于定义“确定性”（如果 A 则 B），但在 AI 时代，我们面对的是“概率性”。AI PRD 最容易掉进的坑就是“许愿”——即只提美好愿景，不写逻辑边界。这份文档存在的意义，就是通过**硬指标（逻辑底线）**与**品味层（表达效率）**的量化评估，确保每一份 PRD 都能让研发敢排期、让测试有考纲、让老板看清 ROI。

---

## 🏗️ 评估架构预览

| 维度分类 | 核心分数 | 评估焦点 |
| :--- | :--- | :--- |
| **Part 1: 硬指标层** | **60 分** | 业务基线、架构解耦、异常与性能边界、评测体系 |
| **Part 2: 品味层** | **40 分** | 信息降噪、读者同理心、决策确定性 |
| **总分** | **100 分** | 默认评分基准，可由 Part 3 动态权重机制覆写 |

---

## 🛡️ Part 1：硬指标层 (60 分)

### 1.1 业务价值与量化目标 (18 分)
**核心逻辑：** 讲清楚“为什么”比“怎么做”更重要。拒绝为了 AI 而 AI。

* **满分档 (18分)：** 给出清晰的历史基线 (Baseline) 与预期提升目标 (Target)，且有合理的逻辑推导。
* **合格档 (13分)：** 有明确量化指标（如：转化率提升 5%），但缺乏基线对比。
* **勉强档 (5分)：** 仅有定性描述（如：显著提高效率），属于随时会扯皮的雷。
* **不合格 (0分)：** 目标完全缺失或强行蹭热点，无实际业务支撑。

### 1.2 架构解耦纯净度 (18 分)
**核心逻辑：** 明确“大脑”与“手脚”的关系。AI PRD 必须严谨区分 Skill 与 Agent，避免把所有能力塞进一个万能黑盒。

* **满分档 (18分)：** **Skill（技能）** 被定义为高内聚、低耦合的模块化文档或功能件；**Agent（智能体）** 被定义为负责整合 Skill 与知识库、进行意图判断和路由调度的核心引擎，并明确输入、输出与调用边界。
* **合格档 (13分)：** 区分了引擎和工具，但数据流转格式（如 JSON Schema 强制约束）描述不全。
* **勉强档 (5分)：** 口头区分了 Agent / Skill / API，但在流程图或接口描述里仍存在职责重叠。
* **不合格 (0分)：** 把 API 调用、数据库查询和业务逻辑全塞在一起，把 Agent 当成“万能大黑盒”。

### 1.3 异常流、性能边界与兜底策略 (12 分)
**核心逻辑：** 处理 AI 的“幻觉”、超时与性能波动是 PM 的基本功。本项将**性能边界（含 P95/P99 延迟约束）**与**功能降级策略**合并评估，因为在真实系统中，性能异常往往就是触发降级的第一信号。

* **满分档 (12分)：** 穷举了接口超时（P95/P99 约束）、敏感词拦截、输出格式破损、幻觉/低置信度等异常节点，并给出具体 UI 降级、重试、人工接管或安全中止方案。
* **合格档 (8分)：** 写了网络超时和基础报错兜底，但忽略了模型特有的异常（如幻觉、格式不遵从、低置信度）。
* **勉强档 (4分)：** 只写了“失败后提示用户重试”，缺少触发条件、降级路径和责任归属。
* **不合格 (0分)：** 只有理想路径（Happy Path），对报错和边界条件只字不提。

### 1.4 评测与验收标准 (12 分)
**核心逻辑：** AI 产品的交付物不是功能，而是评测报告。

* **满分档 (12分)：** 明确评测集规模（N条）、场景分布比例（正常 vs 长尾），并设定数值红线（如 F1-Score > X%、人工抽检通过率 > Y%）。
* **合格档 (8分)：** 有及格红线，但没有规定评测样本分布，容易产生“刷分”虚假繁荣。
* **勉强档 (4分)：** 只有少量人工验收样例，缺少可复用评测集和稳定指标。
* **不合格 (0分)：** 验收标准极度主观（如“生成语气自然”），缺乏量化尺子。

## ✨ Part 2：品味层 (40 分)

### 2.1 信息降噪与结构化表达 (12 分)
**核心逻辑：** 能画图绝不写字。拒绝把 PRD 写成叙事散文。

* **满分档 (12分)：** 状态流转全部流程化（Mermaid/时序图），判定条件全部表格化，关键结论能在 3 分钟内被定位。
* **合格档 (8分)：** 有大纲和列表，但针对复杂模型逻辑仍存在超过 100 字的大段文字描述。
* **勉强档 (4分)：** 结构存在，但信息密度不稳定，读者需要反复回看才能拼出完整逻辑。
* **不合格 (0分)：** “黑话”通稿。充斥着赋能、抓手等词汇，缺乏实际判定逻辑。

### 2.2 读者同理心与视角隔离 (13 分)
**核心逻辑：** 看人下菜碟。让老板看价值，研发看契约，设计看交互。

* **满分档 (13分)：** 具备完美的阅读分层。有 Executive Summary，且技术数据契约、UI 规则、业务收益章节隔离，互不干扰。
* **合格档 (9分)：** 分块清晰，但细节偶有越界（如在给研发看的章节里大谈 Loading 动画的呼吸感）。
* **勉强档 (4分)：** 有分章节意识，但仍主要按作者构思顺序展开，读者需要自己筛选重点。
* **不合格 (0分)：** 纯自我感动视角。完全按照构思顺序写流水账，所有人必须读完全篇才能找准重点。

### 2.3 决策确定性 (15 分)
**核心逻辑：** PRD 是承诺书，不是讨论稿。AI 产品不确定性本就高，PM 再犹豫，团队就会失控。

* **满分档 (15分)：** 所有关键争议点给出 Plan A + Plan B，并明确推荐方案、选择理由、触发切换条件和责任人。
* **合格档 (10分)：** 核心路径已拍板，次要分支允许“待讨论”，但不影响研发排期和测试准备。
* **勉强档 (5分)：** 有倾向性判断，但关键约束仍写成“视情况而定”，团队执行时需要二次确认。
* **不合格 (0分)：** 通篇“待定 / 讨论 / TBD”，把决策成本外溢给研发、设计和测试。

## ⚙️ Part 3：动态权重机制

为防止“一套标准走天下”，根据需求类型自动触发权重调节。默认总分仍为 100 分，但 Part 1 / Part 2 的权重可按需求形态重新映射。

| 需求类型 | 硬指标权重 | 品味权重 | 识别信号 |
| :--- | :--- | :--- | :--- |
| **轻量级 API 接入** | 50% | 50% | 关键词：Prompt 模板、生成文案、单轮对话 |
| **复杂 Agent 工作流** | 75% | 25% | 关键词：知识库、意图路由、工具/Skill 调用 |
| **底层感知/多模态** | 85% | 15% | 关键词：视觉帧率、召回率、P99 延迟、物理约束 |

**触发机制：** 默认由系统 Planner 自动嗅探关键词建议权重，PM 可在评估前一键手动覆写。

**解释原则：** 权重调节不改变 Rubric 的底层维度，只改变不同需求形态下的扣分敏感度。底层感知需求如果不重硬指标就是许愿；轻量级 API 如果过度强调架构解耦就是官僚主义。Planner 的价值，正是在二者之间做上下文判断。

## 📝 笔者自述

这份 Rubric 的整理过程，本质上是对“PM 到底在交付什么”的深度反思。在上海互联网一线做 AI 产品，最痛苦的不是模型效果不好，而是团队在“无效需求”上空转。

我参考了业界主流 Agent 框架的解耦思想，并重点吸收了 Claude Skills 中“Skill 作为可复用能力单元、Agent 负责调度与决策”的设计启发，将其降维到 PRD 的文字要求中。最难权衡的是“品味”的量化，我最终选择放弃纯感性评价，转而用“图表占比”“章节隔离”“是否敢于拍板”这类可观察信号来代表品味。这份文档不希望被供在案头，我希望它能出现在你下一次评审会的投影屏幕上，帮你在面对研发质疑时，能底气十足地说出：“我的逻辑已经闭环了。”

## ⚠️ 本 Rubric 的已知局限

1. **主观维度的量化边界：** Part 2 的“品味”评估依赖 LLM 或评审者判断，存在一定波动。当前版本先用结构化信号降低主观性，后续应通过历史 PRD 样本做校准测试。

2. **需求类型识别的误判风险：** Part 3 的关键词嗅探可能误判边界需求。对于跨类型需求，必须允许 PM 手动覆写权重。

3. **Rubric 本身需要迭代：** 这是 v1.0。随着 AI 产品形态演变，预计 3-6 个月内会出现新的评估维度，例如 Agent 多步推理的可解释性、工具调用链路的可回放性，以及人机协作责任边界。
